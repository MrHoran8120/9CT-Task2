Overview

This assessment task consists of 3 phases. Each phase must be completed sequentially, i.e. Phase 1 must be completed before commencing Phase 2. Each phase is documented on its own page (see navigation bar above). Phases 1 and 2 are peer marked.

The goal of this project is to give students the opportunity to learn about the following key programming concepts in a Python context:

  • Algorithms
  • Loops
  • Conditionals
  • User input
  • Functions
  • Commenting and code readability

Marks and Weighting

Each phase of this assessment contributes a different percentage to the overall mark:

  • Phase 1: 80%
  • Phase 2: 15%
  • Phase 3: 5%

The weighting reflects the significance of each phase. Phase 1 is by far the most important and should be prioritised. Many students in previous years have not had sufficient time to complete Phase 3, so ensure that the earlier phases are finished first.

Phase 1: Reverse Engineering & Menu System

Weighting: 80%

Core Objective

You will be given a Python menu based program that you will be able to run but not read. You must run the program and observe the behavior of the original program and reproduce the output exactly under all conditions, including spacing, punctuation, and capitalization.

Implementation Steps

  1. Standalone Programs (Options 1-4): Develop a separate Python file for each of the first four menu options. These programs must produce the output, including the "Start of Output" and "End of Output" markers.
  2. Menu Integration: Once the first four standalone programs are marked as complete, integrate all required options (1 through X) into a single, comprehensive menu-based system.

Peer Marking Process (Critical)

Immediate Marking:

You MUST have each menu option marked immediately upon completion. Do not "save up" your work for later.

Becoming a Marker:

The first student to successfully complete an option, reviewed and approved by the teacher, earns the title of "Marker" for that specific question.

Marker Responsibility:

Markers review other students' work by running both the model and the student's program simultaneously to confirm exact output matching.

Phase 2: Encrypt / Decrypt (Caesar Cipher)

Weighting: 15%

You will implement a robust Caesar Cipher for text file processing.

Required Files and Tasks (4 Total)

Task File Name Functionality
Encrypt `[StudentNumber]-encrypt.py` Read input file, prompt for offset, write encrypted content to output file.
Decrypt `[StudentNumber]-decrypt.py` Read encrypted file, prompt for offset, decrypt content, write decrypted message to output file.
Known Word Attack `[StudentNumber]-knownValue.py` Deduce the key using one known word in the encrypted text. Output success or failure message.
Brute Force Hack `[StudentNumber]-dictionaryValue.py` Test all 25 offsets against a dictionary file to find the correct key. Output success or failure message.

Phase 3: Game Development

Weighting: 5%

Objective: Write a Simple Game

If time allows, develop a simple, fully functional game in Python.

  • Suggestions: Hangman or a Word Guessing game.
  • Focus: Ensure the game is functional, well-structured, and clearly commented.

Submission Details (Phase 3)

  • File Name (Python): `[StudentNumber]-game.py` (or similar, clearly labeled).
  • ZIP File Name: `[StudentNumber]-Phase3.zip`
  • Structure: The ZIP file must contain only the Python game file(s).

How to Submit & General Rules

General Coding & Submission Rules

Mandatory Code Quality

  • Commenting is mandatory.
  • Comments: All programs, functions, and non-trivial logic must be appropriately commented. Lack of comments will severely reduce your marks (max 4/25 marks awarded).
  • Variables: Variable names should be meaningful and descriptive (e.g., `user_choice` instead of `uc`).
  • Output Match: Output from your program must EXACTLY match the required output from the model program under all conditions.
